Forge Protocol

Classification-Aware AI Development

Balans | November 2025

Executive Summary

Forge Protocol is a framework for building AI systems across security boundaries, developed to solve five critical problems that hinder secure AI development in defense:

  • ✅ Developers with different clearances can collaborate - without leaking secrets
  • ✅ Code can be built and tested - even when specifications are classified
  • ✅ Compliance is provable - automatic validation against standards
  • ✅ Auditing is complete - full provenance from requirements to deployment
  • ✅ Innovation accelerated - no 6-month SÄPO wait for every startup

Sister Protocol: Guardian Protocol handles operational security (deployed AI). Forge Protocol handles development-time security (building AI).

The Challenge: Why Classified AI Development Fails Today

Problem 1: The Clearance Paradox

Scenario: Senior developer (CONFIDENTIAL clearance) receives requirement: "Implement SECRET algorithm from specification doc-00423." Cannot read spec → Cannot implement correctly → System is insecure.

Example (BirdTurret V3, Iteration 2):

Senior Dev:  "Implement fiber-optic FPV detection"
SECRET Spec: [RF signature ABSENT = fiber-optic]
Dev:         [Cannot read spec, guesses RF signature required]
Result:      Algorithm implemented incorrectly, missed 40% fiber-optic threats

Problem 2: The Data Diode Gap

Scenario: AI trained on SECRET data in "Red Cell" (secure environment). Must test performance → Data OUT of Red Cell → Security breach.

Problem 3: Standards Compliance Gap

Scenario: FMV requires proof of EU AI Act compliance. Development team has documentation in Word/PDF → FMV auditor must manually correlate requirements → 4 hours of review per incident.

Problem 4: Provenance Loss

Scenario: Security incident → Investigation needs to know "Who changed this parameter? Why? Was it approved?" → Git log lacks context → Inconclusive investigation.

Problem 5: Startup Innovation Barrier

Scenario: NATO DIANA startup wants to build AI solution for defense → Needs SÄPO approval (€50K, 6 months) for classified development → Cannot iterate quickly → Loses contract to established player.

The Solution: Forge Protocol

Forge Protocol consists of five technical components that solve each development-time problem:

1. Red Cell Architecture (Classified Development Environment)

Secure "sandbox" for classified AI development:

┌─────────────────────────────────────────────────┐
│ RED CELL (Physically Secured Environment)      │
│                                                  │
│  ┌──────────────────────────────────────┐      │
│  │ Data Diode (IN only, never OUT)     │      │
│  │  ↓                                   │      │
│  │  SECRET Specs → Local LLM → Impl    │      │
│  │  ↓                                   │      │
│  │  SECRET Training Data → Model        │      │
│  │  ↓                                   │      │
│  │  Test Results (metrics only) → OUT  │      │
│  └──────────────────────────────────────┘      │
│                                                  │
│  Physical Security:                             │
│  - Locked door (RFID access)                   │
│  - No USB ports (hardware disabled)             │
│  - No WiFi (faraday cage)                      │
│  - Video surveillance                           │
└─────────────────────────────────────────────────┘

How it solves problem 2: SECRET data never leaves Red Cell (data diode IN only), test metrics can be exported (no secrets), developers get feedback for iteration without seeing classified data. Result: BirdTurret V4 fully tested, 0 security incidents.

2. Classification-Aware Development Flows

Developers with different clearances collaborate without leakage.

CONFIDENTIAL developers see filtered specifications with SECRET sections removed. SECRET spec writers work in Red Cell with full detail. Compliance Plugin automatically filters content based on clearance level.

3. Standards-as-Graphs (Queryable Compliance Requirements)

EU AI Act requirements stored as queryable graphs (Neo4j).

Instead of manual PDF review, FMV auditors run Cypher queries for instant compliance validation. Automatic deployment gates prevent deployment without complete evidence.

4. Git Provenance Tracking

Every code change tracked with AI-contribution + human-review.

MCP Git Tool captures: who changed what, why, confidence level, AI vs. human contribution percentage, review cycles, compliance links. Complete context enables rapid incident investigation.

5. Zenoh Data Diode Enforcement

Information flows IN to Red Cell, never OUT (except metrics).

Compliance Plugin (Rust WASM) intercepts ALL Zenoh operations. Whitelist approach: only precision, recall, F1, latency, throughput metrics allowed out. Images, RF profiles, coordinates blocked automatically.

Value for Swedish Industry

For Defense Industry

Problem: Developing AI systems across security boundaries requires expensive, separate teams for each clearance level → Slow iteration, difficult collaboration.

Forge Protocol solution: Senior devs (CONFIDENTIAL) implement from filtered specs, SECRET spec writers work in Red Cell, automatic compliance validation.

Value: Faster time-to-market, lower development costs (one team, different clearances), easier FMV certification.

For Smaller Defense Actors

Problem: Need SÄPO approval (€50K, 6 months) for classified AI development → Cannot compete with established players.

Forge Protocol solution: Share Red Cell environment (Classified Development as a Service), no own SÄPO application, pay-per-use model (€5K/month vs. €50K capex).

Value: Faster innovation (iteration from day 1), lower entry cost, FMV access.

For Defense Research

Problem: Research on classified threats requires secure environment → researchers cannot collaborate with external academics.

Forge Protocol solution: Write SECRET specs in Red Cell, external researchers implement from CONFIDENTIAL specs, Standards-as-Graphs enables publication (share results, not secrets).

Value: Broader collaboration, faster publication, easier EU funding.
